# v4.3 — Staleness & Federation

**Status:** planned — next release
**Predecessor:** [v4.2.0](../../CHANGELOG.md) (shipped 2026-04-24)
**DoD entries:** `docs/release/v4-outcomes.yaml` — `v4.3-stale-detection`, `v4.3-memory-federation`

---

## 1. Why v4.3

v4.2 shipped memory quality controls: sub-budget cap, adaptive top-k, and observation provenance. Two gaps remain that limit memory reliability over time:

1. **Staleness.** Observations point at file paths. When a file is renamed, moved, or deleted, the observation silently becomes misleading — it still ranks and surfaces in packs but its context no longer matches the codebase. There is no signal to the agent that the memory is stale.
2. **Memory stays per-repo.** In a workspace with multiple repos, memory captured in the backend repo is invisible when working in the frontend repo. A team working across repos re-discovers the same knowledge twice.

v4.3 closes both gaps while staying local-first and requiring no new infrastructure.

---

## 2. Non-goals

- **No semantic embeddings** in the staleness detection path. Detection uses git-based file-existence checks and symbol-ID resolution, not similarity scores.
- **No hosted sync service.** Federation uses the workspace's existing `workspace.yaml` and shared git remotes — no separate server.
- **No external service federation** (Slack, Linear, Notion) in this release. The v4 design doc §6.3 describes that as a future adapter; it is out of scope for v4.3 to keep the release focused.
- **No breaking changes** to `ContextItem`, `MemoryHit`, or any MCP tool signatures.

## 3. Staleness detection

### 3.1 Outcome

`memory stale` lists every observation whose `files_touched` contains at least one path that no longer exists in HEAD. `pack --use-memory` surfaces a `"stale": true` flag on affected hits so agents know not to rely on them. `memory prune --stale` removes or archives them.

### 3.2 Detection mechanism

Three staleness signals, checked in order:

| Signal | How detected | Severity |
|---|---|---|
| File deleted | `git ls-files --error-unmatch <path>` fails | `missing_file` |
| File renamed/moved | `git log --follow --diff-filter=R -- <path>` shows a rename | `renamed` — link to new path |
| Observation age | `created_at` > 90 days AND `last_seen_in_pack` is null | `dormant` |

`missing_file` is the only hard-stale signal. `renamed` attempts auto-migration (updates `files_touched` to the new path and emits a warning). `dormant` is informational only — never auto-removed.

### 3.3 CLI surface

```
context-router memory stale
  → lists observations with broken file refs; columns: id, severity, path, age

context-router memory stale --json
  → machine-readable output; same structure as search_memory --json

context-router memory prune --stale
  → removes all missing_file observations; prints count to stdout
  → add --dry-run to preview without deleting
  → add --archive to move to .context-router/memory/archived/ instead of deleting

context-router memory prune --stale --severity renamed
  → only remove renamed observations that could not be auto-migrated
```

### 3.4 Pack integration

When a stale observation is returned by the ranker:

- `MemoryHit.stale` is set to `True`
- `MemoryHit.staleness_reason` contains the signal string (`"missing_file: src/old.py"`)
- A stderr warning is emitted: `WARN: memory hit <id> may be stale (missing_file: src/old.py)`
- The hit is still included — staleness is advisory, not a hard filter — agents decide how to weight it

### 3.5 Stale index warning

If `graph-index` mtime is older than `git log -1 --format=%ct HEAD`, emit at pack time:

```
WARN: index is behind HEAD by <N> commits — run `context-router index` to refresh
```

Suppressed if `--no-stale-warning` flag is set or if `suppress_stale_warning: true` in config.yaml.

## 4. Memory federation

### 4.1 Outcome

When `--workspace` is active, `search_memory` queries memory from all repos declared in `workspace.yaml`. `pack --use-memory --workspace` injects hits from sibling repos into the pack, labeled with `source_repo`. No new infrastructure; federation uses the filesystem paths already known from `workspace.yaml`.

### 4.2 Mechanism

1. `workspace.yaml` already declares the sibling repo paths for cross-repo edge building.
2. `MemoryRetriever` gains an optional `federated_roots: list[Path]` — populated from `workspace.yaml` repos when `--workspace` is active.
3. Each sibling's `.context-router/memory/observations/` is discovered at query time. Only **committed** observations are federated (provenance check from v4.2 applies). Staged/branch_local observations stay local to their repo.
4. BM25+recency scoring runs across all sources combined. Each result carries `source_repo: str` set to the repo name from `workspace.yaml`.
5. Federated hits count against the memory sub-budget (15% cap from v4.2). `memory_hits_summary` gains a `federated` key.

### 4.3 CLI surface

```
context-router search-memory --query "..." --workspace
  → searches memory across all workspace repos; results labeled with source_repo

context-router pack --use-memory --workspace --query "..."
  → injects federated memory hits; memory_hits_summary includes federated count

context-router memory list --workspace
  → lists observations from all workspace repos; grouped by source_repo
```

MCP: `search_memory` gains an optional `workspace: boolean` flag (default `false`). No breaking change.

### 4.4 Isolation rules

| Provenance | Federated? |
|---|---|
| `committed` | Yes — visible across all workspace repos |
| `staged` | No — visible only in the authoring repo |
| `branch_local` | No — visible only on the authoring branch |

This is the same git-model isolation applied in v4.2, extended across repo boundaries. Teammates share committed knowledge; in-progress work stays local.

### 4.5 Failure modes

- Sibling repo has no `.context-router/` directory → skip with `WARN: repo <name> has no memory; skipping federation` to stderr. Pack builds normally.
- Sibling repo's memory directory is unreadable (permissions) → same warning, skip.
- `workspace.yaml` is missing when `--workspace` is passed → same warning as today: falls back to single-repo mode.

## 6. ADRs

**ADR-014: Staleness detection uses git, not content hashing**

Content hashing (compare observation summary to current file) would require LLM calls or cosine similarity — violates the "no API key" constraint. Git-based file existence (`git ls-files`) is fast, offline, and deterministic. Symbol-ID anchoring (from v4 design §6.2) is deferred; file-path staleness covers 80% of the useful signal at zero cost.

**ADR-015: Federation shares committed observations only**

Sharing staged/branch_local observations across repos would expose in-progress work to teammates on unrelated branches. The same git-model isolation that governs within-repo promotion (ADR-012) applies to cross-repo federation. Committed = reviewed and merged = safe to share.

---

## 7. Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| git ls-files is slow on large repos with many observation paths | Medium | Batch all paths in one git ls-files call; cache result for pack session |
| Federation returns too many hits, filling memory sub-budget with cross-repo noise | Medium | Federated hits compete under the same 15% sub-budget cap; local hits rank higher by default (same-repo recency bonus) |
| workspace.yaml is not committed to the sibling repo (it's gitignored by some users) | Low | Emit a clear warning; federation degrades gracefully to single-repo |
| Rename detection via git log --follow is slow on repos with long history | Low | Cache rename map at `workspace sync` time; invalidate on index update |

## 8. Phase C — Ranking quality recovery (v4.2 evaluation follow-ups)

**Background:** v4.2 evaluation (2026-04-24, `results_v42/summary_v42.md`) confirmed v4.2 is still behind v3.3.1 on F1 (0.359 vs 0.577). The gap is concentrated on single-file-GT tasks where the top-5 confidence scores cluster tightly — the 0.6× adaptive top-k floor doesn't fire, so all 5 slots fill regardless of quality. Three targeted fixes close most of the gap.

### 8.1 Adaptive top-k: gradient pruning (P0)

**Problem:** `_ADAPTIVE_TOPK_FLOOR_RATIO = 0.6` only fires when a trailing item is below 60% of the leader. On typical single-file-GT tasks the spread is ~0.52 → 0.46–0.48, so the floor is 0.312 and nothing gets cut.

**v3.3.1 baseline:** v3.3.1 had no explicit top-k cap — it used greedy `confidence/token` budget admission only, which naturally shed low-value items when the budget was tight.

**Root cause (agent investigation 2026-04-24):** `_dedup_by_file()` (added v4.1, `bb950ad`) removes duplicate-file symbols *before* `_enforce_budget()`, breaking the v3.3.1 knapsack mechanism. v3.3.1 never had an explicit top-k cap — the greedy `confidence/est_tokens` budget admission naturally emitted 1–2 items when a single high-confidence file dominated. After dedup, v4.x faces a different candidate set with narrower confidence spread (leader ~0.52, cluster ~0.46–0.50), so the 0.6× floor (= 0.312) never fires.

**Fix:** Two-rule `_apply_adaptive_top_k`. Both run; the more aggressive cut wins.

```python
_ADAPTIVE_TOPK_FLOOR_RATIO: float = 0.6   # existing — keep for wide spreads
_ADAPTIVE_TOPK_PLATEAU_DELTA: float = 0.02 # new: max within-plateau step
_ADAPTIVE_TOPK_ABS_FLOOR: float = 0.45    # new: plateau must be below this absolute threshold
```

Rule 2 implementation: scan `items[1:]`; find the first index `i` where `items[i-1].confidence - items[i].confidence < PLATEAU_DELTA` AND `items[i].confidence < ABS_FLOOR`. All items from `i` onward are trailing plateau — drop them. This targets the common cluster pattern (0.52 → 0.50 → 0.49 → 0.48 → 0.47) that Rule 1 misses.

**DoD:**
```yaml
id: v4.3-adaptive-topk-gradient
threshold: >-
  On the 12-task benchmark, adaptive top-k must fire on ≥6 tasks (up from 3).
  Avg F1 must be ≥0.450 (vs v4.2's 0.359). Avg tokens must stay ≤2,100.
  No regression on tasks that already returned rank-1 GT in v4.2.
```

### 8.2 Fastapi T1 regression: auxiliary path coverage gap (P1)

**Root cause (agent investigation 2026-04-24):** `_AUX_PATH_RE` only matches `scripts?/`. The `docs_src/` tree is a tutorial/example source that never contains the authoritative fix site, but the conditional test-penalty introduced in `efc26d8` (v4.2) stopped penalising test files when a non-test competitor scores below them. `docs_src/` files are that non-test competitor — their BM25 scores on OAuth2-themed queries are high, they are not penalised, and they push `oauth2.py` below rank 5.

**Fix:** One-line change to `_AUX_PATH_RE` in `ranker.py` (line ~175):

```python
# Before (v4.2)
_AUX_PATH_RE = re.compile(r"(^|[\\/])scripts?[\\/]", re.IGNORECASE)

# After (v4.3)
_AUX_PATH_RE = re.compile(
    r"(^|[\\/])(?:scripts?|docs_src|examples?|fixtures|stubs|mocks)[\\/]",
    re.IGNORECASE,
)
```

This applies the same 0.85× auxiliary-file penalty to `docs_src/`, `examples/`, `fixtures/`, `stubs/`, and `mocks/` paths. Canonical source files (e.g. `fastapi/security/oauth2.py`) are unaffected.

**Spec clarification:** align the `_apply_bm25_boost` docstring to match the implementation. The code fires the penalty when `max_non_test_conf >= new_conf`; the docstring/changelog say "skipped when no non-test competitor" (imprecise). The code is correct; the prose is not.

**DoD:**
```yaml
id: v4.3-fastapi-t1-recovery
threshold: >-
  fastapi T1 (oauth2 docstring): GT file (fastapi/security/oauth2.py) must
  appear in top-5. docs_src/security/tutorial003*.py must rank below oauth2.py.
  _is_test_or_script_path('docs_src/security/tutorial003_py310.py') must return True.
```

### 8.3 Implementation order (Phase C)
