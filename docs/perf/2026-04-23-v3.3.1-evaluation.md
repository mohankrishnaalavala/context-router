# context-router v3.3.1 — performance evaluation (2026-04-23)

> This is a **measurement-only** evaluation. No production code was
> changed. Every improvement opportunity (OPP-\*\*) is backed by a
> measurement captured in this document and is left for a follow-up PR.

> **Write location note.** This doc lives at `docs/perf/` because
> `internal_docs/*` is gitignored at the repo root (`.gitignore:225`).
> The original ask in the task brief allowed this fallback location.

---

## 1. Summary

1. **OPP-01 / P0 — Recall is broken for implement / debug / handover
   queries on Python repos.** Across 14 realistic queries on three
   repos, Recall@20 is **21.4 %** (3 / 14). The obvious target file is
   not in the pack at all for 10 / 14 queries, including the exact
   query that motivated the most recent release
   (`"mcp import crash version lookup"` → `apps/mcp-server/src/mcp_server/main.py`
   is a MISS across both `review` and `implement` modes).
2. **OPP-02 / P0 — Contracts boost dominates every pack, regardless of
   query.** On the context-router repo,
   `packages/contracts/tests/test_models.py` appears **42 times as
   separate excerpts** in every `implement` pack I ran, against four
   unrelated queries (MCP, ranking, pack cache, ranking-knapsack).
   Item-level fragmentation plus `source_type="contract"` getting a
   confidence floor of 0.54–0.64 saturates the top slots with
   contract-flavoured test files before lexical relevance even runs.
3. **OPP-03 / P0 — `index` is ~20× slower than it should be on the
   context-router repo, purely because the scanner walks ignored
   directories.** `graph_index.scanner.Scanner.scan` calls
   `_is_ignored` **108,981 times** to index **214 files** on this repo,
   burning **46 s of 58 s** of profiled time inside `_is_ignored`
   itself. A fresh copy of the same code (no `.venv/.git/.claude/worktrees`
   siblings) indexes in **1.1 s**.
4. **OPP-04 / P1 — MCP contract gap: `progressToken` is read from
   `arguments` instead of `params._meta` as the MCP 2024-11-05 spec
   requires.** A spec-compliant client never triggers progress
   notifications; we verified this with a minimal client that follows
   the spec (0 progress frames) vs. one that passes `progressToken` in
   `arguments` (11 progress frames, as designed).
5. **OPP-05 / P1 — Contracts-endpoint extraction re-runs on every pack
   call.** The `api_endpoints` table is empty after `index`, so
   `Orchestrator._load_repo_endpoint_paths` falls back to a full on-disk
   walk (`extract_contracts(root)` → 11,863 files, ~1.1 s) on every
   `pack` call, serialising ~**40 %** of cold pack latency on this
   repo. Comment in `packages/core/src/core/orchestrator.py:1431-1435`
   already acknowledges the gap.

Runner-ups, covered in §5: the FTS tokenizer has no stemming (pack ≠
packs); the knapsack admission criterion systematically prefers many
tiny symbol excerpts over fewer meaningful functions; the indexer has
no analyzer for Markdown so `handover` mode cannot surface
CHANGELOG / RELEASE notes.

### 3.4 Memory retrieval

Six observations captured via `memory capture`. Search timings (wall
clock, includes Python startup ~0.55 s floor):

```
q=mcp crash                                          real=0.67s  hits=1
q=import fallback version                            real=0.62s  hits=1
q=ranking token budget knapsack                      real=0.59s  hits=1
q=contract endpoint boost api                        real=0.62s  hits=0  <-- false neg
q=java typescript analyzer                           real=0.61s  hits=1
q=progress notifications large pack                  real=0.61s  hits=0  <-- false neg
q=totally unrelated dummy query...                   real=0.61s  hits=0  (correct)
```

False-negative 1: `"progress notifications large pack"` returns 0
hits; stored summary is *"added mcp progress notifications for large
**packs**"*. Dropping the final term `"pack"` (singular) → match.
Adding an "s" back → match. Direct FTS reproduction:

```sql
-- misses (singular vs plural):
SELECT * FROM observations_fts WHERE observations_fts
  MATCH 'progress notifications large pack';  -- 0 rows

-- matches:
SELECT * FROM observations_fts WHERE observations_fts
  MATCH 'progress notifications large packs'; -- 1 row
```

The FTS5 virtual table is using the default `unicode61` tokenizer
with no stemming.

False-negative 2: `"contract endpoint boost api"` stored record has
`"contracts boost for api_endpoints consumers"`. Neither `contract` →
`contracts` (plural mismatch) nor `endpoint` → `endpoints` resolves
under `unicode61`. `api_endpoints` is also tokenised as the single
token `api_endpoints` (underscore = word char), not `api` +
`endpoints`.

Latency itself is fine (< 0.1 s of actual DB time beneath ~0.5 s of
interpreter startup).

### 3.5 MCP handshake + tool call roundtrip

Minimal JSON-RPC client over stdio → `.venv/bin/context-router-mcp`:

| Phase | First-call | Warm |
|---|---:|---:|
| `initialize` | 337 ms (fresh server subprocess) · 90 ms (same subprocess 2nd init) | — |
| `tools/list` | 20 ms | 20 ms |
| `tools/call get_context_pack` review | 540–2300 ms | 80–100 ms |
| `tools/call get_context_pack` implement | 135–1023 ms | 80–100 ms |

The order-of-magnitude drop between first and subsequent `tools/call`
on the *same server process* is the pack cache hit path: once a
`(mode, query, project_root)` tuple is cached (look up `pack_cache`
table — 165 rows on the context-router DB), the MCP server returns in
~85 ms end to end. With a fresh random query it's back to ~1 s, which
matches CLI cold-path numbers in §3.3.

#### Progress notifications — partially broken

A client that follows the MCP 2024-11-05 spec sends `progressToken`
inside `params._meta`:

```json
{"method":"tools/call",
 "params":{"name":"get_context_pack",
           "arguments":{...},
           "_meta":{"progressToken":"P1"}}}
```

Against v3.3.1 this **emits 0 `notifications/progress` frames**, even
for a pack that eventually reports `total_est_tokens = 7972`
(well above the default 500-token threshold in
`apps/mcp-server/src/mcp_server/main.py:925`).

The server reads the token from `arguments.progressToken`
(`main.py:1094`):

```python
progress_token = arguments.pop("progressToken", None)
```

Sending `progressToken` inside `arguments` instead of `_meta` → 11
progress frames arrive, exactly as the unit tests in
`apps/mcp-server/tests/test_progress.py` exercise.

Additional asymmetry: when a pack is served from `pack_cache`, the
orchestrator never fires `progress_cb`, so the `_ProgressGate` buffer
stays empty and `flush_if_large` emits zero. An agent that awaits
"progress=N ⇒ done" will hang on cache hits. Treat as a P2 protocol
nit.

The headline number is the **21× slowdown** on context-router's real
workspace vs. a cleaned-up copy of the same 210 files. `cProfile` of
the full `index` (58 s under profiler overhead; `duration_seconds`
field reports 56.8 s):

| Function | Cum. time | Calls |
|---|---:|---:|
| `graph_index.scanner.Scanner.scan`           | 54.7 s | 215 |
| └─ `Scanner._is_ignored`                      | **46.0 s** | **108,981** |
| `pathlib.Path.relative_to`                    | 28.9 s | 108,981 |
| `fnmatch.fnmatch` (inside `_is_ignored`)      | 11.0 s | 4,213,669 |

Root cause is in `packages/graph-index/src/graph_index/scanner.py:48`:

```python
for path in sorted(self._root.rglob("*")):
    ...
    if self._is_ignored(path):
        continue
```

`rglob("*")` walks into `.venv/` (6k+ packages), `.git/`,
`.claude/worktrees/` (47 sibling checkouts on this machine),
`dist/`, etc., generating ~109 k paths, **then** filters one by one
using `fnmatch` against each path component. The correct pattern is
`os.walk` with in-place `dirs[:] = [d for d in dirs if not ignored]`
so ignored subtrees are never descended. `pathspec` (gitignore-style)
would also be dramatically faster than N × M `fnmatch` calls.

Other indexing observations:

* Stderr is noisy: 123 debug lines reading *"[language-python] debug:
  no in-file SUT for test file ..."*. That's mechanically correct
  (cross-file SUT linking happens in `link_tests`), but a
  developer-facing default.
* Only the Python analyzer registers meaningful symbols on
  `context-router` (`SELECT DISTINCT language FROM symbols` →
  `external`, `python`). YAML / TS / Java / .NET are stubs (files
  present in `packages/language-*/tests/test_stub.py`), which is
  consistent with OPP-07.

## 4. Findings

### 4.1 What's good

* **Review-mode packs are consistently small and token-efficient** —
  ~260–320 tokens on all three repos, ~97 % reduction vs. naive on
  non-trivial ones. This matches the story
  `summary_v330.md` told for v3.3.0 and has not regressed.
* **Pack-cache speed-up is real.** Warm `tools/call` on MCP drops
  from ~1 s to ~85 ms; that's a legitimate and defensible optimisation
  for interactive agents.
* **Graph generation is fast and bounded** (< 1 s for repos tested).
  Memory is bounded (~100 MB).
* **Memory capture and search latencies are within budget** (< 1 s
  wall, < 0.1 s DB). No performance issue there.
* **`external_dropped` metadata is emitted** (6–27 per pack) — the
  opaque `<external>` symbols are correctly filtered before output.
* **`ship-check` silent-failure discipline is visible in the code** —
  e.g., `packages/ranking/src/ranking/ranker.py:524` emits a warning
  before falling back, as §3.2 of CLAUDE.md requires.

### 4.2 What's bad

* **Implement / debug / handover mode recall is unusably low for
  exploratory queries** (§3.2). Users who say
  *"help me implement OAuth2 password bearer"* will not see the
  OAuth2 source module in their pack.
* **Ranking is dominated by `source_type="contract"` fragmentation**
  (§3.1). A single test file can occupy 40+ pack slots. The knapsack
  admission in `packages/ranking/src/ranking/ranker.py:1255-1277`
  sorts by `confidence / est_tokens`, which systematically rewards
  the smallest, most numerous items.
* **Cold indexing on a real workspace is 20× slower than necessary**
  (§3.7). The scanner walks the entire disk before filtering.
* **MCP progress notifications silently do not fire for
  spec-compliant clients** (§3.5) — the server reads the wrong JSON
  field. No stderr warning, no test catches a spec-compliant
  `_meta` path.
* **FTS tokenisation breaks singular/plural recall on memory search**
  (§3.4). Missing `memory search` hits are user-facing.

### 4.3 What's ambiguous

* The `summary_v330.md` claim that context-router rank-1 hits the GT
  file on all 3 fastapi tasks does not reproduce today (§3.2). I
  cannot tell from this eval whether that was a measurement flaw
  then, a regression now, or different flags / DB state. Worth a
  regression-hunt PR of its own, but I do not have enough to assert.
* Semantic ranking (`--with-semantic`) sets `metadata.boosted_items`
  but the top-5 order was identical to keyword-only on the one task I
  compared. Either the boost isn't large enough to move ranks, or the
  embeddings table is empty (`SELECT COUNT(*) FROM embeddings → 0`)
  and the fallback path is inert. This needs a dedicated
  measurement.
* Items-vs-unique-files in pack output: `selected_items` is
  symbol-level. External docs and README language about
  "files\_included" should either reflect that or the serialiser
  should group by file. Not clearly a "bug", but a UX inconsistency.

## 5. Improvement opportunities

### OPP-01 — Low recall on exploratory implement / debug queries · **P0**

* **Observation.** Recall@20 = 21.4 % across 14 queries on 3 repos
  (§3.2). The obvious target is absent from the pack entirely in
  10 / 14 cases, including the precise query that names the bug the
  release fixed (`"mcp import crash version lookup"` ⇒
  `apps/mcp-server/src/mcp_server/main.py`).
* **Hypothesis.** Two compounding causes:
  1. Lexical BM25 in
     `packages/ranking/src/ranking/ranker.py:687-720` is applied
     *after* contract annotation, and contract items already carry a
     high base confidence (≥ 0.54) that lets them win the
     value-per-token knapsack (lines 1245-1277) without a strong
     lexical signal.
  2. Semantic boosting either runs against an empty
     `embeddings` table (no vectors pre-computed by `index`) or the
     ±bonus is too small to reorder. `SELECT COUNT(*) FROM embeddings`
     was 0 on all three test DBs.
* **Proposed fix.** (a) Require a non-zero BM25 floor before any
  item can be admitted to the top-K slice, or invert the tie-breaker
  so `BM25 > 0` items beat contract-flagged items with zero lexical
  overlap. (b) Make `--with-semantic` a visible silent-failure case:
  if `embeddings` is empty, either auto-compute on the fly (warn) or
  print a clear stderr line so users know the flag is inert.
* **Risk.** Over-indexing on lexical tokens could regress the cases
  where a task description doesn't literally name its target
  (handover, cross-module refactor). Need a held-out recall suite so
  this is measured, not argued.
* **Effort.** M — ranker changes are localised but need a new
  honest-recall benchmark to prove the win.

### OPP-02 — Source-type fragmentation in packs · **P0**

* **Observation.** `contracts/tests/test_models.py` shows up 42 times
  per `implement` pack on context-router, independent of the query.
  `fastapi/openapi/models.py` shows 20 excerpts on two unrelated
  fastapi queries. `mockServiceWorker.js` shows 9 excerpts on every
  bpr implement pack — and it's a **generated file**.
* **Hypothesis.** Three compounding causes:
  1. `_enforce_budget` (ranker.py:1255-1277) admits by
     `confidence / est_tokens`. Tiny symbol excerpts (~50 tok each)
     dominate. A 20-symbol test module fits easily.
  2. There is no per-file item cap; the `is_first_of_type` guard
     only ensures *at least one per source_type*, never *at most N
     per file*.
  3. Generated files (`mockServiceWorker.js`) are indexed with no
     special downweight.
* **Proposed fix.** Cap `items_per_file ≤ 3` (or `≤ ceil(sqrt(N))`)
  after knapsack admission; downweight paths matching a new "generated"
  heuristic (checked into the file header or a gitignore-style
  pattern); prefer fewer-but-larger function bodies over many-tiny
  symbol headers.
* **Risk.** Capping may drop a legitimate test-sweep pack for
  review mode; the cap should be mode-aware
  (review tolerates, implement enforces).
* **Effort.** M — logic change is local to `_enforce_budget`, but
  again needs a regression benchmark.

### OPP-03 — Scanner walks ignored subtrees before filtering · **P0**

* **Observation.** `graph_index.scanner.Scanner.scan` calls
  `_is_ignored` 108,981 times on a `context-router` index (§3.7);
  46 s of 58 s profiled time is spent inside that single method.
  Indexing a clean `/tmp` copy of the same 210 Python files: 1.1 s.
* **Hypothesis.** `packages/graph-index/src/graph_index/scanner.py:48`
  uses `self._root.rglob("*")`, which recursively expands every
  directory including `.venv/`, `.git/`, `.claude/worktrees/` before
  any filter runs. Each path is then re-checked with N × M
  `fnmatch` calls (N patterns, M path components).
* **Proposed fix.** Replace `rglob` + per-path fnmatch with `os.walk`
  + in-place `dirs[:] = [...]` so ignored directory subtrees are
  never descended. Alternatively, adopt `pathspec` for gitignore-style
  matching (one compiled object, O(1) per path).
* **Risk.** Small — `os.walk` semantics match `rglob` for file
  discovery. Tests in `packages/graph-index/tests/` should catch any
  ordering drift. Need to re-verify `sorted(...)` behaviour if any
  downstream code relies on deterministic ordering.
* **Effort.** S — ~30-line change in one file + a unit test that
  asserts the skip behaviour (create a fake `.venv/` with many
  files; assert indexing wall-clock scales with **real** file count,
  not tree size).

### OPP-04 — MCP `progressToken` is read from `arguments`, not `_meta` · **P1**

* **Observation.** Spec-compliant client passing `_meta.progressToken`
  gets zero progress frames; putting the same token in `arguments`
  yields 11 frames (§3.5).
* **Hypothesis.** `apps/mcp-server/src/mcp_server/main.py:1094` reads
  `arguments.pop("progressToken", None)`. The inputSchema at
  `main.py:233` also declares `progressToken` as a top-level argument.
  Both are non-spec; the MCP 2024-11-05 contract places the token
  on `params._meta`.
* **Proposed fix.** Read `params._meta.progressToken` first
  (spec-correct path), fall back to `arguments.progressToken` for
  backwards compatibility (emit a stderr warning when the fallback
  triggers — silent-failure discipline, per CLAUDE.md §3.2). Remove
  `progressToken` from the published inputSchema.
* **Risk.** Breaking existing clients that ship progressToken in
  arguments. Mitigation: warn for one release, remove next release.
* **Effort.** S — two-line read + schema update + a new test in
  `apps/mcp-server/tests/test_progress.py` that uses the
  spec-compliant `_meta` shape.

### OPP-05 — `api_endpoints` fallback re-runs on every pack · **P1**

* **Observation.** `_apply_contracts_boost` spends 1.15 s per cold
  pack (40 % of total) because `_load_repo_endpoint_paths`
  (`packages/core/src/core/orchestrator.py:1426-1476`) finds the
  `api_endpoints` table empty and walks the repo with
  `extract_contracts(self._root)` every time. The code comment at
  line 1432 already acknowledges this: *"The single-repo CLI `index`
  command does not yet populate this table — running the parser at
  boost time keeps the feature working."*
* **Hypothesis.** `index` never calls `extract_contracts`; the
  orchestrator does, on the pack hot path. The fallback was intended
  as temporary.
* **Proposed fix.** Add a contracts-extractor step to `index` so
  `api_endpoints` is populated once, and keep the fallback only as a
  degraded-mode last resort (with a stderr warning — silent-failure
  rule). Trivial to make incremental because the `extract_contracts`
  output is per-file.
* **Risk.** Index time increases slightly; mitigate by respecting
  the existing ignore patterns.
* **Effort.** M — add an `on_file` hook in `Indexer.run` to feed
  `ContractRepository.bulk_upsert`; retire the on-disk fallback for
  the `api_endpoints` source after a deprecation notice.

### OPP-06 — FTS5 tokenizer has no stemming · **P1**

* **Observation.** `memory search "progress notifications large pack"`
  returns 0 hits against a summary containing *"progress notifications
  for large packs"* (§3.4). Similar breakage for
  `endpoint` ↔ `endpoints`.
* **Hypothesis.** The FTS5 virtual tables `observations_fts` and
  `decisions_fts` use SQLite's default `unicode61` tokenizer with no
  `porter` pipeline. `SELECT sql FROM sqlite_master WHERE name LIKE
  'observations_fts'` shows no explicit tokenizer.
* **Proposed fix.** Add `tokenize = 'porter unicode61'` on table
  creation. Requires a one-time migration (rebuild the FTS tables;
  data in `observations`/`decisions` itself is untouched).
* **Risk.** Porter is English-only; non-English observations will
  still work but not stem. Acceptable for our users.
* **Effort.** S — migration script; the tables are already small
  (6 rows on this DB).

### OPP-07 — Non-Python indexing on mixed repos is thin · **P2**

* **Observation.** `bulletproof-react` indexing produced 268 nodes /
  58 edges — an edges-per-node ratio of 0.22 vs. ~1.5 for Python
  repos. TypeScript analyser lives in `packages/language-typescript/`
  but its functional coverage is visibly low on a real-world TS
  repo.
* **Hypothesis.** The TS analyzer in
  `packages/language-typescript/src/language_typescript/` uses
  tree-sitter but extracts symbols without import/call edges, or
  uses a restricted subset of node types. Could not verify in detail
  in this eval.
* **Proposed fix.** Expand the TS analyser to emit `imports` and
  `calls` edges via tree-sitter. This also lifts recall on bpr from
  2 / 4 to something meaningful.
* **Risk.** Larger code change; TS has many call / import syntactic
  variants (dynamic `await import()`, re-exports, `as` aliases).
* **Effort.** L — this is a feature-level lift, not a one-commit
  fix.

### OPP-08 — Markdown analyzer missing, `handover` mode can't surface docs · **P2**

* **Observation.** `CHANGELOG.md` and `RELEASE.md` are never indexed
  (`SELECT DISTINCT language FROM symbols` → `external`, `python`),
  so a `handover` pack asked about a recent release cannot find
  them.
* **Hypothesis.** No analyzer plugin is registered for the `.md`
  extension. `handover` mode relies on runtime evidence + symbols
  but has no docs-surface ingestion path.
* **Proposed fix.** A minimal Markdown analyzer that emits one
  `symbol` per heading (level 1 & 2), with the heading text as the
  name, kind=`section`. That's sufficient for BM25 to rank
  `CHANGELOG.md` / `RELEASE.md` / `docs/release/*` in handover mode.
* **Risk.** Adds potentially-noisy files to the index; mitigate via
  mode-aware filtering.
* **Effort.** M — new analyzer package, new language entry, index
  migration.

### OPP-09 — Silent no-op on `--with-semantic` when `embeddings` is empty · **P2**

* **Observation.** `--with-semantic` completes successfully with
  `metadata.boosted_items` populated but the same item ordering as
  keyword-only (§3.2). The `embeddings` table has 0 rows on all three
  test DBs.
* **Hypothesis.** `_apply_semantic_boost` encodes the query but has
  no stored item vectors to compare, falls back to zeros
  (`packages/ranking/src/ranking/ranker.py:524-530`), and still
  emits the boosted metadata even though no reordering happened.
* **Proposed fix.** If `len(stored_vectors) == 0` after the fetch,
  print a stderr warning
  (*"--with-semantic requested but no pre-computed embeddings;
  ranking unchanged — run `context-router embed` or set
  `CONTEXT_ROUTER_PROACTIVE_EMBED=true`"*). Already aligned with the
  silent-failure rule.
* **Risk.** None material — this is purely an observability fix.
* **Effort.** S — three lines in `ranker.py`.

### OPP-10 — Pack-cache hit skips progress notifications · **P2 / speculative**

* **Observation.** On an MCP `tools/call` with progressToken, if the
  pack is served from `pack_cache`, the orchestrator never calls
  `progress_cb`, so `_ProgressGate.flush_if_large` emits zero
  notifications (§3.5). An agent awaiting `progress=total`
  before treating the response as complete will never see it.
* **Hypothesis.** The progress-callback path is wired only into the
  full build path in `Orchestrator.build_pack`; the cache hit path
  returns early.
* **Proposed fix.** Emit a single `notifications/progress` with
  `{progress: total, total: total, message: "cached"}` right
  before the `tools/call` response when serving from cache.
* **Risk.** None.
* **Effort.** S.

### OPP-11 — Index stderr spam on every test file · **P2 / speculative**

* **Observation.** Indexing context-router prints 123 stderr lines
  of `[language-python] debug: no in-file SUT for test file ...`.
  Indexing fastapi prints > 500.
* **Hypothesis.** The Python analyzer emits a debug message for
  every test file whose SUT will be linked cross-file; the default
  log level lets these through.
* **Proposed fix.** Demote to `trace` level or gate behind
  `CONTEXT_ROUTER_DEBUG=1`.
* **Risk.** None.
* **Effort.** S.

### OPP-12 — D3 CDN fetch in generated graph HTML · **informational**

`apps/cli/src/cli/commands/graph.py` (not modified by this PR) emits
`https://d3js.org/d3.v7.min.js`, making the graph unusable offline.
A sibling agent is landing a fix; I verified the presence but did not

## 6. Non-findings

Things that looked suspicious at first but turned out to be fine:

* **Warm pack latency** (~85 ms through MCP) is genuinely good; I
  initially flagged it as "suspiciously fast", but it's a legitimate
  `pack_cache` hit, not a skipped build.
* **`external_dropped`** metadata of 6–27 per pack was initially
  read as "data lost". It's the correct ranker behaviour —
  `<external>` placeholder items have no resolvable file path and
  should not be returned to the client. A stderr note is emitted;
  behaviour matches `note: dropped N opaque <external> placeholder
  item(s) from pack`.
* **Memory search "contract endpoint boost api" → 0 hits** looks like
  a ranking problem but is actually tokenization (plural/underscore)
  — consolidated into OPP-06 rather than a separate opp.
* **Graph HTML size (1.6 MB on fastapi)** felt large at first but is
  just `JSON.stringify` on 4,939 nodes; compresses to ~300 KB over
  the wire. Not worth optimising.
* **Peak RSS of 303 MB during `index`** on the real workspace looked
  alarming. It correlates with the scanner sweeping `.venv/.git/`
  and temporarily holding many `Path` objects. Fix OPP-03 will
  eliminate this too.
* **bpr baseline 13,368 tokens** making `implement` packs only 50 %
  reduction — the tool is doing the right thing; the repo is genuinely
  small. Not a bug.
* **`review` mode "item=5 tokens=311"** triggering alarm bells — this
  is by design (`review-mode defaults applied` stderr note). Not a bug.
