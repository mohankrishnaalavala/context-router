# context-router v4.4.2 — precision follow-ups

> Patch release on top of v4.4.1. Closes the rank-1 regression introduced by `--with-rerank` and ships the two phases that were deferred from the v4.4 plan: query-conditional feedback (Phase 6) and the docs-only-diff heuristic (Phase 7). Single PR; three slices.

## Headline

The judge agent's evaluation of v4.4.1 surfaced a regression: **`--with-rerank` was promoting test files above production sources** on tasks where the test name quoted the query verbatim (fastapi T1, eShop T1). On fastapi T1 the production source `oauth2.py` fell out of the pack entirely.

v4.4.2 closes that rank-1 regression with a small multiplicative pass after the cross-encoder blend, and lands the deferred work that was already documented in `docs/superpowers/plans/2026-04-27-performance-enhancements.md` under "Future phases."

## What's in this release

### 1. Rerank source-prior fix (closes rank-1 regression)

After the existing 0.5/0.5 cross-encoder/structural blend, multiply the blended score by a multiplicative source-preference prior:

- **source files**: `blended *= 1.15`
- **test/aux files**: `blended *= 0.85`
- clamp to `[0.0, 0.95]`

Reuses the existing `_is_test_or_script_path()` classifier (matches `tests/`, `__tests__/`, `spec/`, `*_test.*`, `*Test.*`, `*Spec.*`, `mocks/`, `fixtures/`, `scripts/`, `examples/`, `stubs/`, `docs_src/`).

The prior is a **tiebreaker, not a veto** — when the cross-encoder strongly disagrees (e.g. test +10 raw / source -10 raw → sigmoid 1.0 vs 0.0), the test still wins. The fix only restores the source/test asymmetry when the cross score is roughly neutral, which is exactly when fastapi T1 / eShop T1 went wrong.

### 2. Phase 7 — docs-only-diff heuristic

v4.4 Phase 3's query-driven candidate widening is suppressed whenever `changed_files` is non-empty, on the assumption that a real diff is the authoritative signal. PRs whose entire diff is a doc bump (e.g. fastapi T1 / T3 in the tuning benchmark have a tiny `release-notes.md` change) suppress widening exactly when widening would help.

New classifiers `_is_docs_path()` and `_is_docs_only_diff()` recognize:
- File suffixes: `*.md`, `*.rst`, `*.adoc` (case-insensitive).
- Top-level / docs-path filename prefixes: `release-notes*`, `CHANGELOG*`, `LICENSE*`, `CONTRIBUTING*`, `CODE_OF_CONDUCT*`, `SECURITY*`, `HISTORY*`, `NOTICE*`, `AUTHORS*`, `MAINTAINERS*`, `README*`.
- Directory names: any path with a `docs/`, `documentation/`, or `doc/` segment.
- `*.txt` only counts as docs when it lives under a docs directory or matches a top-level marker prefix.

When the entire diff is docs, the widening gate flips to fire — `query_match` candidates surface and the GT can recover. **Mixed PRs (any code + docs) keep the v4.4.1 authoritative-diff behaviour.** The `changed_files` set itself is unchanged, so diff-aware boost and `changed_file` source-type continue to see it.

### 3. Phase 6 — query-conditional feedback

v4.4 Phase 4 closed the feedback loop: after ≥3 reports for a file, `missing` adds +0.05, `noisy` -0.10, `files_read` +0.03. But adjustments were per-file *globally* — a file flagged `noisy` once took the −0.10 penalty on every future query, even orthogonal ones.

Phase 6 cosine-weights each historical adjustment by query similarity:

- **Schema migration `0014`** adds two nullable columns to `pack_feedback`: `query_text TEXT NULL` (raw query string for debugging) and `query_embedding BLOB NULL` (compact MiniLM-384 float32 vector ≈ 1.5 KB per row).
- **Write path** — `FeedbackStore.add()` opportunistically embeds the query via the existing bi-encoder (`ranking._get_embed_model()`). Silent-degrades to NULL when sentence-transformers is unavailable.
- **Read path** — `get_file_adjustments()` accepts a new `current_query_embedding: bytes = b""` kwarg. When non-empty, the per-row delta is multiplied by `max(0, cosine(current, row))`. Negative cosines clamp to 0 — opposing queries contribute nothing rather than flipping the sign.
- **Aggregation** — `delta = base_delta * (sum(cosine_weights) / raw_count)`. The `min_count=3` raw threshold still gates whether an adjustment fires; weighting only modulates magnitude.
- **Backwards compatibility** — legacy rows with NULL `query_embedding` AND calls with `current_query_embedding=b""` both produce the v4.4.1 unweighted result. No behaviour change for existing data or callers.

The orchestrator now opportunistically embeds the current query before calling `get_file_adjustments`, so existing call sites benefit automatically.

## Trade-offs / known limitations

1. **The rerank prior is multiplicative.** A 0.85× penalty on a 0.10 test takes it to 0.085, possibly below the score floor — that's the intended effect (kill bad test promotions), not a bug.
2. **Docs-only classifier is conservative.** Auto-generated files (`*.pb.go`, `*_pb2.py`, `Designer.cs`) are NOT docs; mixed PRs (1 doc + 1 code) keep authoritative-diff behaviour. The fastapi T1 / T3 cases are 100% docs and will fire.
3. **Phase 6 needs feedback rows with embeddings to materialise its value.** Existing legacy rows contribute their full delta (no penalty); only NEW feedback rows recorded after upgrade get the embedding column populated. The behaviour migrates gracefully as the test environment generates feedback.
4. **Negative cosines clamp to 0.** An orthogonal historical query contributes nothing rather than flipping the sign of the adjustment. Conservative — avoids "noisy in query X means useful in query Y" feedback inversions.
