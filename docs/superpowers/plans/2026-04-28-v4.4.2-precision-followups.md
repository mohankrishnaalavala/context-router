# context-router v4.4.2 — precision follow-ups (rerank fix + Phase 6 + Phase 7)

## Context

v4.4.1 shipped (commit `ba95107`, tag `v4.4.1`, GitHub release published). The 12-task tuning benchmark vs `code-review-graph` lifted F1 to **0.583** (best v4.x ever, tied with v3.3.0's 0.577) at **1,906 avg tokens** (best of any v4.x release). However, the judge agent's evaluation surfaced one regression and two known unsolved gaps:

| Issue | Detail |
|---|---|
| **Rank-1 dropped to 7/12** (worst since v4.0) | `--with-rerank`'s 0.5/0.5 cross-encoder/structural blend lets the cross score override the source-preference prior. Test files with high lexical overlap get promoted above the production source on **eShop T1** and (worse) **fastapi T1** drops `oauth2.py` out of the pack entirely. |
| **fastapi T1 / T3 unsolved** | These are doc-only-diff PRs. Phase 3's query-driven widening short-circuits when *any* changed_file exists — the trivial `release-notes.md` diff suppresses widening. |
| **Per-file feedback bleeds across queries** | A file flagged `noisy` once gets the −0.10 penalty on every future query, even orthogonal ones. |

v4.4.2 closes the rank-1 regression first (highest-leverage agent UX fix), then ships Phases 6 and 7 from the deferred-work list in `2026-04-27-performance-enhancements.md`.

## Approach — three slices, one PR

All three slices ship in **one PR** (per user instruction) on branch `release/v4.4.2`. Each slice is implemented by a parallel agent in an isolated worktree; the slices are merged into the release branch and the full suite runs once.

### Slice A — Rerank source-prior fix (rank-1 regression)

**Where:** `packages/ranking/src/ranking/ranker.py` — inside `_apply_cross_encoder_rerank` at the end (after the 0.5/0.5 blend).

**Algorithm:**
1. After the blended `final_conf` is computed for a reranked item, classify the item's path:
   - **source** if the path is *not* matched by `_AUX_PATH_RE` and is *not* matched by the existing test-path regex used by the v4.1 test penalty.
   - **test/aux** otherwise.
2. Apply a multiplicative source-prior:
   - source files: `final_conf *= 1.15`
   - test/aux files: `final_conf *= 0.85`
3. Clamp to `[0, 1]`.

**Why multiplicative-after-blend (not blend-time):** the cross score should still reorder *within* a class (source vs source, test vs test), but a test should never outrank a same-relevance source. Multiplicative-after-blend preserves cross's value while restoring the structural prior's veto.

**Tests:** new file `packages/ranking/tests/test_rerank_source_prior.py`:
- `test_source_prior_promotes_source_over_test_with_equal_cross_score` — synthetic items with identical cross scores; source ends ranked above test.
- `test_source_prior_does_not_invert_huge_cross_gap` — when cross-encoder scores a test 0.4 above the source, the test still wins (prior is a tiebreaker, not a veto).
- `test_source_prior_clamps_to_one` — applied to a 0.95-confidence source doesn't overflow.
- `test_source_prior_classification_uses_aux_re` — `tests/`, `__tests__/`, `*.spec.ts`, `*.test.py`, `mocks/`, `fixtures/` all classified as test/aux.

### Slice B — Phase 7: docs-only-diff heuristic

**Where:** `packages/core/src/core/orchestrator.py` — inside `_review_candidates`, at the point where `query_tokens = _tokens_for_query_match(query) if not changed_files else set()` is computed.

**Algorithm:**
1. Define `_DOCS_ONLY_PATTERNS`: `*.md`, `*.rst`, `release-notes.*`, `CHANGELOG*`, `docs/**`, `documentation/**`, `*.txt` only when in a docs path.
2. Helper `_is_docs_only_diff(changed_files: list[Path]) -> bool`:
   - Returns `True` only if the list is non-empty AND every entry matches a docs pattern.
   - Returns `False` if any code file is touched, or if the list is empty.
3. Replace the widening gate:
   - **Before:** `query_tokens = _tokens_for_query_match(query) if not changed_files else set()`
   - **After:** `query_tokens = _tokens_for_query_match(query) if (not changed_files or _is_docs_only_diff(changed_files)) else set()`
4. Do **not** change the `changed_files` list itself — it still drives `changed_file` source-type candidates. Only the widening gate sees the docs-only override.

**Tests:** new file `packages/core/tests/test_phase7_docs_only_diff.py`:
- `test_docs_only_diff_classifier_recognises_md_rst_changelog` — happy path.
- `test_docs_only_diff_classifier_rejects_mixed_diff` — one .py + one .md → `False`.
- `test_widening_fires_on_docs_only_diff` — orchestrator integration: `changed_files=["release-notes.md"]` + a code-flavoured query → `query_match` source_type appears in candidates.
- `test_widening_suppressed_on_code_diff` — `changed_files=["src/main.py"]` → no `query_match` candidates.
- `test_empty_diff_still_widens` — Phase 3 baseline preserved.

### Slice C — Phase 6: query-conditional feedback (cosine-weighted adjustments)

**Where:**
- Schema: `packages/storage-sqlite/src/storage_sqlite/database.py` — `pack_feedback` table.
- Write path: `packages/memory/src/memory/store.py` — `FeedbackStore.add` and / or `PackFeedbackRepository.add`.
- Read path: `packages/storage-sqlite/src/storage_sqlite/repositories.py` — `get_file_adjustments`.
- Surface: `packages/core/src/core/orchestrator.py` — `feedback_applied` metadata builder.

**Algorithm:**
1. **Schema migration.** Add two nullable columns to `pack_feedback`:
   - `query_text TEXT NULL` — raw query string (for debugging).
   - `query_embedding BLOB NULL` — compact float32 vector (use `numpy.array.tobytes()`; ~1.5 KB per row for MiniLM-384).
   Use `ALTER TABLE … ADD COLUMN IF NOT EXISTS` pattern (or check `PRAGMA table_info` and conditionally ALTER) to keep migrations idempotent and backward-compatible.
2. **Write path.** When a feedback row is recorded with a query attached, compute the embedding lazily via the existing `_get_embed_model()` helper. If the model is unavailable (silent-degrade contract), store `NULL` and the row falls back to the v4.4.1 unweighted behaviour.
3. **Read path.** `get_file_adjustments(min_count=3, *, current_query_embedding: bytes | None = None)`:
   - When `current_query_embedding is None`, behaviour is **identical** to v4.4.1 (no regression).
   - When provided, sum `cosine(current_query_emb, row_query_emb) * delta` across rows with non-NULL embeddings. Rows with NULL embedding contribute their full delta (legacy behaviour, no penalty).
   - Threshold check: `min_count` still counts raw rows (3+ reports gate stays the same).
4. **Surface.** Extend `pack.metadata.feedback_applied` entries from `{"path", "delta"}` → `{"path", "delta", "query_weight"}` where `query_weight` is the avg cosine across contributing rows (or `1.0` if all rows were NULL-embedding legacy rows).

**Tests:** extend `packages/memory/tests/test_feedback.py`:
- `test_query_embedding_persisted_when_available` — feedback row stores BLOB.
- `test_query_embedding_null_when_model_unavailable` — silent-degrade: store NULL, no exception.
- `test_get_file_adjustments_unweighted_when_no_current_embedding` — v4.4.1 behaviour preserved.
- `test_get_file_adjustments_cosine_weights_similar_query_higher` — synthetic embeddings: `[1, 0]` (current) vs `[1, 0]` (history) gives full delta; `[0, 1]` vs `[1, 0]` gives ~0.
- `test_legacy_rows_with_null_embedding_unweighted` — mixed table: NULL rows count at full delta, BLOB rows weighted.

## Trade-offs called out

1. **Slice A is multiplicative, not additive.** A 1.15× boost on a 0.95-confidence source goes to 1.0 (clamped) — fine. But a 0.85× penalty on a 0.10 test takes it to 0.085, possibly below the score floor. That is the *desired* effect (kill bad test promotions), not a bug.
2. **Slice B's docs-only classifier is conservative.** Auto-generated files (`*.pb.go`, `*_pb2.py`, `Designer.cs`) are *not* docs-only — touching them legitimately means code changes. Mixed PRs (1 doc + 1 code) keep their authoritative-diff behaviour. The fastapi T1 / T3 cases are 100% docs and will fire.
3. **Slice C migration is forward-compatible.** Existing rows have NULL `query_embedding`; legacy reads return identical results to v4.4.1. New rows start contributing weighted signal as soon as the test environment generates feedback with queries attached.
4. **Rollback plan.** Each slice is a self-contained patch; reverting one of the three by file diff is straightforward if the test environment shows an unexpected regression on a specific slice.
