# v3.3.0 — First‑run works, and MCP streams

Status: approved (2026-04-19)
Owner: primary session
Target release: v3.3.0

## Problem

Real‑world eval (CR vs code‑review‑graph, 2026‑04‑19) found three shipping‑quality problems that dwarf any remaining algorithmic gap:

1. **Fresh installs index zero files.** Language‑analyzer plugins published to PyPI / Homebrew don't register entry points on clean installs, so `context-router index` finds no analyzers and emits an empty graph. The tool looks broken on first try.
2. **Default pack is unusable.** Emitting ~50k tokens with the right answer buried in 100+ items makes the ranker's actual accuracy (GT at rank 1 in 12/12 with sane flags) invisible. `token_budget: 8000` in `config.yaml` is silently ignored.
3. **`<external>` placeholders poison precision.** Opaque rank‑2 entries with no file path eat tokens and eval metrics.

Two additional items came back signal‑positive in the eval:

4. MCP streaming + resources — real value for Claude Code users on large packs.
5. Agent‑friendly output mode — `[{path, lines, reason}]` vs prose excerpts.

## Scope (v3.3.0)

| Lane | Items | Why it's here |
|---|---|---|
| **α** Packaging | α1: language‑plugin dist‑info in published wheels · α2: clean‑venv smoke test | Without this, nothing below matters |
| **β** Ergonomics | β1: honor `token_budget` from config · β2: flip `--top-k 5 --max-tokens 4000` default for `--mode review` · β3: resolve / filter `<external>` · β4: `--format agent` output · β5: ranking TTLCache at orchestrator scope | The eval says ergonomics + defaults, not the algorithm, is the gap |
| **γ** MCP | γ1: progress notifications in `get_context_pack` · γ2: `resources/list` + `resources/read` for prior packs · γ3: pack registry in `.context-router/packs/` | Closes the Claude Code UX gap on large packs |
| **δ** Docs | δ1: mode docs (when to use review/debug/implement/handover) | Judge called out missing guidance |

**Explicitly out of scope:** function‑level call graph (P3‑4), cross‑language contracts extractor (P3‑3). Deferred to v3.4 or v4.0 once real users ask for them.

## Work breakdown

Four lanes, three launch as parallel worktree agents, δ handled inline.

**Lane α — packaging** (`backend-agent`, worktree)
- Repro zero‑index bug by building a wheel from each `packages/language-*/` and pip‑installing into a clean venv.
- Fix: ensure `[project.entry-points."context_router.language_analyzers"]` survives `hatch build` and lands in the wheel's `*.dist-info/entry_points.txt`.
- Add `scripts/smoke-packaging.sh` that rebuilds + installs into a tmp venv + asserts entry points resolved.
- `doctor` subcommand gains an "analyzer entry points" check.

**Lane β — core/CLI ergonomics** (`backend-agent`, worktree)
- `CapabilitiesConfig` reads `token_budget` → pass through `build_pack()` default → fail loud when overridden.
- `--mode review` at `apps/cli/commands/pack.py`: if `--top-k` not provided, default = 5; if `--max-tokens` not provided, default = 4000. Emit a one‑line stderr advisory the first time per session.
- `<external>` resolution in `packages/core/src/core/orchestrator.py`: try to map to real path; drop if unresolved; count via `pack.metadata.external_dropped`.
- New `--format agent` in pack CLI → list of `{path, lines, reason}` via `pack.to_agent_format()`.
- Add `cachetools.TTLCache` (maxsize 100, ttl 300) on orchestrator; key = `(repo_id, mode, query_hash, budget, items_hash)`; bust on `update_index`/`build_index`.

**Lane γ — MCP streaming + resources** (`backend-agent`, worktree)
- `_notify()` helper + mutex on stdout writes in `apps/mcp-server/src/mcp_server/main.py`.
- Capabilities block gains `resources: {listChanged: true}` and `progress: true`.
- `PackStore` at `packages/core/src/core/pack_store.py` — persists last 20 packs to `.context-router/packs/<uuid>.json`.
- `resources/list` + `resources/read` handlers at `apps/mcp-server/src/mcp_server/resources.py`.
- Progress emit points in `get_context_pack`: after candidate build, after ranking, after serialization (3 frames total) — only when output ≥ 2000 tokens.

**Lane δ — mode docs** (inline)
- `docs/guides/modes.md` — one paragraph per mode, a decision tree at the top, and a table mapping "I am trying to …" → mode.
- Link from README.

## Integration / release flow

1. Each lane agent opens a PR to `develop` with tests + ship‑check verdict pasted.
2. Primary session reviews + merges each lane PR serially.
3. Primary session bumps all 20 `pyproject.toml` files to `3.3.0`, updates CHANGELOG, appends v3.3 entries to `v3-outcomes.yaml`, runs `scripts/smoke-v3.sh report`.
4. Open release PR `develop → main`; wait for CI green.
5. User merges; tag `v3.3.0`; release workflow publishes to PyPI + Homebrew tap.

## Risks

- **Hatchling packaging fix could be non‑trivial** if the plugins' build isolation loses entry points — fallback is switching language plugins to a single pyproject with multiple entry points.
- **MCP async stdio interleaving** — mitigated by a `threading.Lock` guarding every `sys.stdout.write`.
- **TTLCache correctness under MCP server** — single writer today, but the cache needs to be per‑session not global to avoid cross‑repo pollution.
- **Homebrew tap PAT still outstanding** — orthogonal; blocks release publish only, not merge.
