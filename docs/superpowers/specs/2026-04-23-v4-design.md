# v4 — Multi-repo awareness and team-shareable memory

**Status:** design — pending review
**Author:** @mohankrishnaalavala + Claude
**Last updated:** 2026-04-23
**Predecessor:** [v3.3.0 design](./2026-04-19-v3.3.0-design.md) (shipped 2026-04-20 as v3.3.1)

---

## 1. Why v4

v3 stabilised the single-repo story — indexing, ranking, MCP packaging, and shipping discipline. The product is now good at one repo. Two external-judge reviews and one multi-repo eval (`fastapi` vs `code-review-graph`) agreed on the same two gaps:

1. **Multi-repo awareness.** Modern services are split across frontend, backend, and one or more shared schemas. context-router currently treats each repo as an island, so a "change the checkout flow" query against the frontend can't surface the backend handler that actually owns the state.
2. **Team-shareable memory.** Observations and decisions live in per-user SQLite. Two engineers on the same team rediscover the same tribal knowledge; an agent on a fresh clone has none. Judges and users flagged this as the single biggest lift between "useful for me" and "useful for us."

v4 addresses both, sequenced so each release is demo-able and safe to ship on its own.

| Release | Theme | User can now... |
|---------|-------|-----------------|
| **v4.0** | Multi-repo measurement | Build a pack whose candidate set spans a `workspace.yaml` of repos, and read an honest Recall@K number for their fixture. |
| **v4.1** | Memory-as-code (foundation) | `save_observation` writes a git-tracked markdown file; `--use-memory` on `pack` and `get_context_pack` retrieves prior learning. |
| **v4.2** | Memory promotion & sub-budget | `git commit` on a memory file is the promotion signal; pack ranker uses no more than 15% of the token budget for memory. |
| **v4.3** | Semantic staleness & federation | Memory entries anchored to symbol IDs survive rename/move; optional Slack / Linear federation exposes external artefacts through the same ranker. |

The scope of each release is locked to one visible outcome. The judge concerns below are mapped to phases so we can point at the commit that resolved each one.

### 1.1 Judge concerns mapped to phases

| Concern (yesterday's review) | Phase |
|---|---|
| "Multi-repo: what happens when my frontend and backend are separate repos?" | **v4.0** |
| "You claim multi-repo but your eval is single-repo." | **v4.0** |
| "Cross-service links should come from real contracts, not a hand-written workspace.yaml." | **v4.0** (protobuf/GraphQL), **v4.3** (external systems) |
| "Memory is per-user; teammates can't share what the agent learned." | **v4.1** + **v4.2** |
| "Git notes are the obvious sync mechanism." | Rejected in favour of git-tracked files — see ADR §7.1 |
| "Memory growth means agents waste tokens on stale or irrelevant observations." | **v4.1** (selective write + cached retrieval), **v4.2** (sub-budget), **v4.3** (symbol-ID anchoring) |

## 2. Non-goals

v4 explicitly does **not** do any of the following. Naming them avoids scope creep and makes the design reviewable.

- **No hosted sync service.** Memory sharing happens through the repo's own git remote, not a separate context-router server.
- **No semantic embeddings in the hot path for memory retrieval.** v4.1/v4.2 retrieval is BM25 + recency. Embeddings stay opt-in (v3.3 `--with-semantic`).
- **No cross-org telemetry.** Everything is local-first. We collect nothing.
- **No MCP protocol changes.** v4 reuses the `get_context_pack` / `search_memory` surface; new flags only.
- **No auto-migration of existing SQLite observations into files.** A one-shot CLI command handles this when the user asks.
- **No benchmark score in the README blog yet.** v4.0 is "measurement only" — we publish a number but do not tune against it until v4.1+ lands.

## 3. v4.0 — Multi-repo eval-first

### 3.1 Outcome

A user with a multi-repo project declared in `workspace.yaml` can run `context-router pack --query "..." --workspace` and receive a pack that includes the most relevant file from *each* repo in the workspace, weighted by explicit contract links (OpenAPI / protobuf / GraphQL) and implicit import analysis. On the public benchmark fixture, Recall@20 must reach **0.65** — a number we commit to only after we have measured the honest baseline.

### 3.2 Why eval-first

v3 shipped three P0 bugs because we measured code, not outcomes. For a feature as subjective as "the right file from the right repo," the gate has to be a reproducible number, not a code review. Eval-first means:

1. Build the fixture and the metric first.
2. Freeze today's recall number as the baseline (pre-v4.0).
3. Land the feature only when the CI-enforced Recall@K crosses the threshold.

### 3.3 Fixture: synthetic + 3 real

Hybrid fixture lives in `tests/fixtures/workspaces/`:

- **`synthetic/`** — 2 tiny repos (Python backend + TypeScript frontend) with a hand-crafted OpenAPI contract between them. Used by CI on every PR; always green, ~2s. Catches regressions without network or large clones.
- **`real/spring-petclinic-microservices/`** — cloned on demand by `scripts/fetch-benchmark-repos.sh`. Java microservice suite; already part of v3 benchmarks. Exercises protobuf/REST mix.
- **`real/realworld/`** — React frontend + Node backend pair; community benchmark. Exercises JS↔TS↔REST flow.
- **`real/saleor/`** — large monorepo+satellite with GraphQL schema. Exercises our new GraphQL parser.

CI runs the synthetic fixture. The three real fixtures run nightly (or on `--eval` opt-in locally) and publish numbers to a `eval_results.jsonl` artifact checked into the release tag only.

### 3.4 Metrics

Two metrics, split by purpose:

| Metric | Role |
|---|---|
| **Recall@20 on queries from `queries.jsonl`** | Hard gate. CI fails if recall drops below the frozen baseline OR below the v4.0 target of 0.65 on the synthetic fixture. |
| **Token-efficiency (Recall@20 / pack_tokens × 1000)** | Narrative metric for the release blog. Reported but not gated. |

Queries and gold-standard answers are hand-written per fixture and version-controlled in the same folder. Example: `tests/fixtures/workspaces/synthetic/queries.jsonl` contains `{"q": "add rate limiting to checkout", "gold": ["backend/api/checkout.py", "frontend/src/checkout/index.tsx"]}`.

### 3.5 Contract extraction

The `packages/contracts-extractor/` package (scaffolded in v3 Phase P3 Lane D but currently empty) gets three parsers:

- `parsers/openapi.py` — `.yaml` / `.json`, produces `ApiEndpoint(method, path, operation_id, …)`.
- `parsers/protobuf.py` — `.proto` files; signatures only, no semantics.
- `parsers/graphql.py` — `.graphql` / `.gql`; produces `GraphqlOperation(name, kind, fields)`.

These feed a new `api_endpoints` / `grpc_services` / `graphql_operations` set of tables in storage-sqlite (migration `0011_contracts.sql`). The `workspace_orchestrator` uses a `consumes` edge from "repo B contains a request to path X" to "repo A exposes path X as an endpoint" as the primary cross-repo ranking signal.

### 3.6 Scope call-outs (the "b" option, locked)

- **In scope:** parser trio, contract tables, workspace-aware orchestrator, Recall@K harness, synthetic+3 real fixtures, `--workspace` CLI flag, `context-router eval` subcommand.
- **Out of scope for v4.0:** ranker tuning against the eval, UI/docs polish beyond a short "multi-repo" guide, semantic similarity of contracts. v4.0 is measurement + plumbing; knob-turning is a follow-up that gets gated by the same eval.

### 3.7 DoD (fills `docs/release/dod-template.md`)

```yaml
dod:
  outcome: >-
    `context-router pack --workspace --query "<query>"` produces a pack whose
    candidate set spans every repo declared in workspace.yaml, with contract
    edges (OpenAPI/protobuf/GraphQL) used for ranking.
  threshold: >-
    Recall@20 ≥ 0.65 on tests/fixtures/workspaces/synthetic/queries.jsonl AND
    not lower than the frozen pre-v4.0 baseline on each real-world fixture.
  negative_case: >-
    Running --workspace in a repo with no workspace.yaml emits
    "warning: --workspace has no effect; no workspace.yaml found at <path>"
    to stderr and falls back to single-repo mode with exit 0.
  verify:
    cmd: >-
      context-router eval --fixture synthetic --metric recall@20 --json
    expected_stdout_contains: '"recall_at_20": 0.'
```

Companion DoDs for the parser trio and the `eval` subcommand live alongside the main outcome in `docs/release/v4-outcomes.yaml` (skeleton in §8).

## 4. v4.1 — Memory-as-code foundation

### 4.1 Outcome

`save_observation` (and `save_decision`) writes a git-tracked markdown file under `.context-router/memory/` with YAML frontmatter. `pack`, `get_context_pack`, and `search_memory` retrieve those files through the existing BM25 ranker, gated by a new `--use-memory` flag. A teammate who `git pull`s now has every observation the first engineer saved, with zero new infrastructure.

### 4.2 Why files, not git notes

The v3→v4 team-memory literature review (captured in memory obs `#2355–#2361`) is unanimous: by Q1 2026, every production system that reached >1k stars had converged on **git-tracked markdown files in a dedicated directory** — Anthropic (Nov 2025), Letta MemFS (Feb 2026), Memorix, Oxford GCC (arXiv:2508.00031). The argument against git notes:

- Notes are a parallel ref tree most devs never see. PR diffs and code review don't cover them, so team oversight is nil.
- Tooling (GitHub, GitLab, editors, grep) does not surface notes naturally.
- They can't be refactored with the code they describe.

Files win on every axis we care about: diff-review, searchability, CI coverage, refactor-safety, editor-native.

ADR §7.1 captures this decision.

### 4.3 Storage layout

```
.context-router/
  memory/
    observations/
      2026-04-23-checkout-dedup.md
      2026-04-23-mcp-import-fix.md
      …
    decisions/
      0001-git-notes-rejected.md
      0002-memory-budget-15pct.md
    MEMORY.md        # auto-regenerated index (one line per observation)
  context-router.db  # SQLite cache; derived, gitignored
```

File format:

```markdown
---
id: 2026-04-23-checkout-dedup
type: observation           # observation | decision
task: debug                 # implement | review | debug | handover
files_touched:
  - packages/core/src/core/orchestrator.py
symbols_touched:            # v4.3 turns this into symbol-ID references
  - orchestrator.build_pack
created_at: 2026-04-23T13:58:00-05:00
author: mohankrishnaalavala
---

# Dedup in the pack table happens at write, not read

Problem: `Pagination` row appeared 3× for the same file path. …

(Free-form markdown. The ranker tokenises title + body.)
```

SQLite stores a derived index (id → path, files_touched, mtime, content_hash). `context-router index` rebuilds it idempotently; file-system watch keeps it warm during a session.

### 4.4 Selective write (token-cost floor)

The judge's concern — "more memory files means agents spend more tokens reading memory" — is answered by never letting the agent read raw files. Four quality gates live between "a thought" and "a memory":

1. **Write gate: relevance filter in `save_observation`.** The server rejects obs under 60 chars, or obs whose `files_touched` is empty, or obs marked `type: scratch`. Principle: a memory must point at code.
2. **Retrieval gate: BM25 + recency rank, top-K=8.** The agent never sees more than 8 obs per query, and only the excerpted snippet the ranker picked, not the whole file. At a conservative 200 tokens/obs × 8 = 1.6 kt ceiling on memory spend.
3. **Sub-budget (v4.2).** Memory tokens may not exceed 15% of the pack's total token budget. Enforced in `packages/ranking/src/ranking/ranker.py`.
4. **Staleness eviction (v4.3).** Obs whose anchor symbols were deleted or renamed drop in rank. Three-strike eviction: three consecutive pack builds that excluded an obs move it from `memory/observations/` to `memory/observations/archive/`.

This is the full answer to "does memory consume more tokens as it grows?" — no. The ceiling is fixed and enforced at both retrieval and ranking.

### 4.5 CLI surface

New flags, no removals:

```
context-router pack --use-memory --query "…"
context-router pack --memory-budget 0.15         # defaults baked in v4.2
context-router memory list                       # human-readable
context-router memory show <id>
context-router memory migrate-from-sqlite        # one-shot, v4.1 ships this
```

MCP `get_context_pack` gains an optional `use_memory: boolean` arg; schema update, no breaking change.

### 4.6 DoD

```yaml
dod:
  outcome: >-
    `save_observation` writes a git-tracked .md file under
    .context-router/memory/observations/, and `pack --use-memory` includes
    the top-ranked observations in the pack output.
  threshold: >-
    After `save_observation(summary="…", files_touched=["X.py"])`, a file
    matching .context-router/memory/observations/*-*.md exists whose
    frontmatter lists "X.py". `pack --use-memory` with a matching query
    includes the observation's id in its candidate list within the first 8
    entries.
  negative_case: >-
    `save_observation(summary="hi")` (no files_touched, <60 chars) is
    rejected with "refused: observation must reference at least one file"
    on stderr and exit 2.
  verify:
    cmd: >-
      context-router pack --use-memory --mode implement
        --query "checkout dedup" --project-root <fixture> --json
    expected_stdout_contains: '"memory_hits":'
```

## 5. v4.2 — Promotion & sub-budget

### 5.1 Outcome

A memory file is "promoted" (counted as part of shared team knowledge) only after it's committed to the repo's shared branch. Uncommitted or branch-only observations rank normally for the author but are filtered out for anyone on a different branch — the standard git model applied to memory. The 15% sub-budget from §4.4 becomes a hard cap.

### 5.2 Why promotion matters

Without a promotion step, every `save_observation` immediately affects a teammate's next pack. That creates two failure modes: (a) a half-baked debug note from Alice fires into Bob's pack and distracts him; (b) the shared memory trends toward noise because there's no human review.

Git commit is the right gate: it's already the team's quality bar for anything else in the repo. PR review covers memory diffs the same way it covers source diffs.

### 5.3 Mechanism

- `save_observation` writes the file and `git add`s it (stage only — does not commit).
- `pack` picks up staged + committed files from the current branch.
- `search_memory` and `get_context_pack` from a clean checkout of `main` pick up only committed-to-main files. Unmerged work never leaks into a teammate's pack.
- `memory_hits` in pack metadata reports `{committed: N, staged: M, branch_local: K}` so the user knows the provenance.

### 5.4 Sub-budget enforcement

Ranker change (`packages/ranking/src/ranking/ranker.py`):

```python
memory_token_cap = int(total_budget * config.memory_budget_pct)  # default 0.15
memory_items = self._rank_memory(candidates, memory_token_cap)
code_items = self._rank_code(candidates, total_budget - sum(i.tokens for i in memory_items))
```

Cap is configurable in `.context-router/config.yaml`; default is 0.15 (15%).

### 5.5 DoD

```yaml
dod:
  outcome: >-
    Committed memory files surface in teammate packs; uncommitted files
    do not. Memory tokens never exceed 15% of pack budget.
  threshold: >-
    (a) Pack built on clean `main` after `git pull` includes at least one
    obs committed by a teammate. (b) `pack_tokens_memory / pack_tokens_total
    ≤ 0.15` on every release-fixture query.
  negative_case: >-
    If memory_budget_pct is set ≤ 0 or ≥ 1 in config.yaml, context-router
    emits "warning: memory_budget_pct out of range, using 0.15" to stderr
    and falls back to the default.
  verify:
    cmd: >-
      context-router pack --use-memory --json --query "…" | jq '.budget.memory_ratio'
    expected_stdout_contains: "0."
```

## 6. v4.3 — Staleness & federation

### 6.1 Outcome

Memory entries survive refactors: an obs that points at `orchestrator.build_pack` still surfaces after the function moves to a new file, because we anchor on symbol ID, not file path. Optional federation lets a user register a Slack channel, Linear project, or Notion space in `workspace.yaml`; those artefacts rank through the same `search_memory` path, without leaving the context-router CLI.

### 6.2 Symbol-ID anchoring

Today the `files_touched` / `symbols_touched` frontmatter is free text. In v4.3:

- Symbols get a stable ID (`repo_id::module::symbol::signature_hash`) stored in `symbols` table (already present).
- `save_observation` resolves the symbol name to an ID at write time.
- On index refresh, renamed/moved symbols retain their ID; observations follow them automatically.
- Deleted symbols mark their obs as `stale: true` in the derived SQLite; pack ranker down-weights them.
- Three-strike rule (§4.4.4) archives observations whose all anchors are stale.

### 6.3 Federation (optional, feature-flagged)

A thin adapter pattern:

```
packages/federation/
  src/federation/
    slack.py      # reads an export or a bot token; yields ChannelMessage
    linear.py     # GraphQL; yields LinearTicket
    notion.py     # page tree; yields NotionBlock
    base.py       # Federator ABC
```

Adapters are off by default. Turning on Slack in `workspace.yaml`:

```yaml
federation:
  slack:
    workspace_export: ~/exports/my-team.zip  # or bot_token: env:SLACK_BOT_TOKEN
    channels: [ "#eng-platform", "#incidents" ]
  linear:
    team_id: ENG
    api_key: env:LINEAR_API_KEY
```

Every adapter produces items with the same `MemoryCandidate` shape the ranker already consumes; nothing downstream changes.

### 6.4 DoD

```yaml
dod:
  outcome: >-
    An obs anchored to a symbol still ranks for the same query after the
    symbol is renamed or moved, without any user action. Federation, when
    enabled, surfaces Slack/Linear items in `search_memory` results.
  threshold: >-
    After a benchmarked `git mv` + symbol rename sequence, an obs keyed
    to the renamed symbol still appears in `search_memory` top-8 for the
    original query. Federation adapter smoke test: with a 100-message
    fixture Slack export, `search_memory --federation slack` returns the
    expected thread by its ts.
  negative_case: >-
    Federation misconfigured (bad token, missing export path) emits
    "warning: slack federation disabled: <reason>" to stderr and the
    pack still builds without federation items.
  verify:
    cmd: >-
      context-router memory search --query "import crash" --json
    expected_stdout_contains: '"anchor_id":'
```

## 7. ADRs to record

### 7.1 Why git-tracked files, not git notes

Locked on 2026-04-20 after review of Memorix, Memori, Letta MemFS, Anthropic guidance, and Oxford GCC. Reasons (short form):

- **Discoverability.** Files show up in PRs, GitHub search, and `grep`. Notes do not.
- **Refactor-safety.** A memory file can be renamed with the code it describes. A note cannot without re-writing.
- **Team review.** PR diffs cover memory changes. Notes require a separate workflow.
- **Tooling ecosystem.** Every editor, CI, and static analysis tool understands markdown files. Almost none understand notes.

We lose the "auto-attached to a commit" property of notes; we replace it with frontmatter `anchor.commit_sha` on each obs, which is more precise.

### 7.2 Why 15% memory sub-budget

Arbitrary round number, but constrained by: (a) top-K=8 × 200 tokens/obs ≈ 1.6 kt ceiling, (b) default pack budget is ~12 kt, so 15% ≈ 1.8 kt — the two numbers meet. Anything lower and retrieval becomes too aggressive; anything higher and code context starts getting evicted.

### 7.3 Why "measurement-only" v4.0

We have no honest Recall@K number today. Tuning before measuring produces cargo-cult wins. v4.0 publishes the number; v4.1+ tunes against it with the number as the gate.

### 7.4 Multi-repo graph topology — one per repo, plus a thin workspace edge DB

The natural question when introducing `workspace.yaml` is: do we build **one graph across all repos** or **one graph per repo stitched together at query time**? We are picking the second, with a caveat.

**Decision:** keep today's per-repo `.context-router/context-router.db`, unchanged. Add a new, small `<workspace_root>/.context-router/workspace.db` that stores only:

- A repo registry (repo_id, repo_root, last_indexed_at, schema_version).
- A cross-repo edge table: `(src_repo, src_symbol_id, dst_repo, dst_symbol_id, edge_kind, confidence)` where `edge_kind ∈ {consumes_openapi, consumes_grpc, consumes_graphql, import, workspace_declared}`.
- A contract registry (OpenAPI / protobuf / GraphQL artifact pointers).

This is the hybrid option, not a "giant unified graph."

**Why not one big graph:**

- **Blast radius.** One corrupt repo would poison everyone. Today a bad index is scoped to one repo; we want to keep that.
- **Atomic updates.** Per-repo writes are cheap and already work with the file-system watcher. A single-DB design forces global locking.
- **Partial clones.** A teammate who has only checked out the backend should still get useful single-repo packs; a global DB would make that awkward.
- **Release velocity.** The per-repo DB schema is stable. A unified schema re-write would be a much bigger change than v4.0 should try to ship.

**Why not strictly per-repo:**

- Cross-repo ranking requires joins across edges that span repos. Computing them ad-hoc at every `pack --workspace` means N-way SQLite queries; slow and messy.
- A tiny edge DB that materialises exactly these cross-repo joins keeps `pack --workspace` a single SQL query per ranking pass.

**Auto-update when the frontend changes:**

1. Frontend's existing in-repo watcher (v3.3) catches the change and updates the **frontend's** `.context-router/context-router.db` — unchanged behaviour.
2. The workspace layer subscribes to "a repo's DB mtime changed" events. When a repo bumps, the workspace `reconcile` pass (see below) runs for **that repo only**, not the whole workspace.
3. `reconcile` re-runs the contract parser and link-detector on the changed repo, emits its current set of `(src_symbol, dst_symbol, edge_kind)` rows, and upserts them into `workspace.db` (delete-and-replace scoped to `src_repo=<changed>`). Symmetric `dst_repo=<changed>` rows are matched by symbol-ID resolution, not by full rescan of siblings — constant work even when repos are large.
4. A workspace-level watcher polls per-repo DB mtimes every 5s in `watch` mode, or runs once on every explicit `context-router pack --workspace` (mtime check is cheap; no rebuild if every repo DB is older than the workspace DB).

**Triggers, explicit vs implicit:**

- **Implicit (default):** every `pack --workspace` query checks per-repo DB mtimes. If any is newer than the workspace DB, reconcile that repo inline. Latency budget: ≤50ms per changed repo; the bulk of the pack-build cost is still ranking, not reconcile.
- **Explicit:** `context-router workspace sync` — force-rebuild the workspace edge table. Useful after a big refactor or on first setup.
- **Daemon (v4.3 optional):** `context-router workspace watch` — a single long-running process that multiplexes file-system watchers across every repo in the workspace and keeps the edge DB hot. Feature-flagged; not required for v4.0.

**Failure modes:**

- If a repo in the workspace has no index at all (user never ran `init` in it), `pack --workspace` emits `warning: repo <X> not indexed; skipping` and proceeds with the rest. No hard failure.
- If the workspace DB schema is older than the CLI's expected version, the CLI drops and rebuilds it (it is a pure cache).
- If two repos claim the same `repo_id` (collision or misconfig), CLI errors out explaining the fix — don't silently merge.

**What this means for the design doc (§3.5):** the contract tables (`api_endpoints`, `grpc_services`, `graphql_operations`) live in `workspace.db`, not the per-repo DB. The per-repo DBs only hold what they hold today.

**What the user sees:** nothing new in single-repo flows. In workspace flows, opening `graph.html` (post-v4.0) shows nodes coloured by their source repo, with dashed cross-repo edges. Implementation note captured for the writing-plans handoff; not gated by v4.0 DoD.

## 9. Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Synthetic fixture under-represents real-world complexity → false-green CI | High | Nightly real-fixture run against 3 OSS repos; floor CI on real-fixture recall at "must not regress." |
| `.context-router/memory/` directory becomes a 10k-file dumping ground | Medium | Selective-write + three-strike archive; hard limit of 2k active obs per repo (archive oldest on overflow). |
| Protobuf/GraphQL parsers are big surface for the v4.0 budget | Medium | Parse signatures only; skip semantics. If a grammar is missing, emit warning and degrade to "contracts not available for this repo." Fall back to v3 import-based link detection. |
| Agent reads raw memory files and burns tokens | Low | Retrieval is always via ranker, never raw file read. Document this loudly. Add an integration test that greps MCP stdout for memory content larger than the top-K snippet size. |
| Teammate leaks branch-local obs into shared memory | Low | Promotion via git commit is the gate; staged-only obs show `provenance: staged` in metadata so consumers can filter. |

## 10. Rollout
