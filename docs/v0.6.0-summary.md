## What was built

### P4 — BM25 query scoring

**File:** `packages/ranking/src/ranking/ranker.py`

Replaced exact substring matching (`_apply_query_boost`) with inline Okapi BM25 (`_apply_bm25_boost`).

- Formula: `final_conf = min(0.95, 0.6 × structural_conf + 0.4 × bm25_score)`
- BM25 scores are normalized per-query across all candidates (most relevant = 1.0)
- ~50 lines of pure Python — no new dependency (k1=1.5, b=0.75)

**Why inline, not a library:** avoids adding `rank-bm25` or scikit-learn as a wheel dependency; BM25 core is simple enough to maintain inline.

**Behavior change:** items with no query match now get `0.6 × structural_conf` (not 100%). High-confidence structural items (e.g. `changed_file` = 0.95) drop to 0.57 if not query-relevant — this is intentional; it prevents infrastructure files dominating the pack when the query targets application logic.

### P5 — Call flow analysis

**Files:**
- `packages/storage-sqlite/src/storage_sqlite/repositories.py` — `EdgeRepository.get_call_chain_files()`
- `packages/core/src/core/orchestrator.py` — `_debug_candidates()` now appends `call_chain` items
- `apps/cli/src/cli/commands/explain.py` — `--show-call-chains` flag

BFS traversal of `calls` edges (already stored from Python/TypeScript/C# analyzers) up to depth 3.  
Confidence decay: depth 1 = 0.45, depth 2 ≈ 0.315, depth 3 ≈ 0.22 (×0.70 per hop).

**Why debug mode only:** call chains are most useful when surfacing code paths that lead to an error site. Review/implement modes already have blast_radius from import edges which covers structural adjacency. Adding call chains there would increase noise.

**Cycle safety:** the BFS tracks `visited_ids` (symbol IDs) so cycles in the call graph terminate cleanly.

### P6 — `files_read` on PackFeedback

**Files:**
- `packages/contracts/src/contracts/models.py` — `files_read: list[str]` field
- `packages/storage-sqlite/src/storage_sqlite/migrations/0007_pack_feedback_files_read.sql`
- `packages/storage-sqlite/src/storage_sqlite/repositories.py` — `PackFeedbackRepository.add()`, `_row_to_feedback()`, `aggregate_stats()`
- `apps/cli/src/cli/commands/feedback.py` — `--files-read` option
- `apps/mcp-server/src/mcp_server/tools.py` — `record_feedback(files_read=...)`
- `apps/mcp-server/src/mcp_server/main.py` — inputSchema updated

Read coverage analytics activate after ≥ 5 reports include `files_read`:
- `read_overlap_pct`: fraction of files_read that were NOT in the noisy list (useful reads)
- `noise_ratio_pct`: fraction of files_read that were in the noisy list

**Why this matters:** Without `files_read`, we can only infer quality from missing/noisy signals. With it, future analysis can answer "what fraction of what the agent actually used did context-router include?" — the real precision metric.
